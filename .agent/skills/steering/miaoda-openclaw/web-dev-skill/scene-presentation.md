# 演示文稿场景开发规范（presentation）

> **本文档是演示文稿场景的权威开发流程规范。** 严格按照下方定义的两个阶段逐步执行，不可跳过或合并任何步骤。

---

**核心定位 - 扩充为完整演示**：

所有单一开发需求都必须扩充为完整的演示文稿：

| 用户输入 | 扩充为 | 包含内容 |
|---------|-------|---------|
| "做个PPT" | 完整演示文稿 | 封面页 + 内容页 + 结尾页 |
| "做个培训演示" | 培训演示文稿 | 按培训逻辑组织页面 |
| "做个产品路演" | 产品路演文稿 | 按说服逻辑组织页面 |

## Phase 1: 准备阶段

> 产出物：`AGENTS.md`（包含应用概览 + 需求拆解 + 风格设计）。**AGENTS.md 写入完成前禁止进入 Phase 2。**

### Step 1: 需求分析

<thinking>

##### PS-1. 内容理解与受众分析

**内容理解**：
- **核心主题**：这份演示要传达什么核心信息？
- **内容来源**：基于什么输入材料？（文档/数据/知识点）
- **内容体量**：信息量大小，预估演示时长

**受众分析**：
- **目标受众**：谁会观看？（管理层/客户/学员/公众）
- **观看场景**：在什么环境观看？（会议室/线上/大屏幕）
- **期望收获**：观众看完后应该记住/理解/行动什么？

##### PS-2. 演示类型判定

根据内容和目的判断演示类型，指导后续结构设计：

| 演示类型 | 判断依据 | 结构特点 | 节奏特点 |
|---------|---------|---------|---------|
| **培训演示** | 教授知识/技能 | 循序渐进，由浅入深 | 稳定，留有消化时间 |
| **产品路演** | 说服/展示价值 | 痛点->方案->价值->行动 | 有起伏，强调亮点 |
| **会议汇报** | 传递信息/决策 | 结论先行，数据支撑 | 高效，重点突出 |
| **课程讲解** | 引导理解概念 | 概念->示例->练习->总结 | 循环，反复强化 |

##### PS-3. 演示结构设计（核心）

**结构设计原则**：
- 每页一个核心观点，避免信息过载
- 章节之间有逻辑关系（递进/并列/对比）
- 开头吸引注意，结尾强化记忆

**大纲规划**：

基于输入内容，设计演示大纲：

| 页面序号 | 页面类型 | 核心内容 | 信息要点（2-4个） |
|---------|---------|---------|-----------------|
| 1 | 封面页 | 演示标题 | 主标题、副标题、元信息 |
| 2 | ... | ... | ... |
| N | 结尾页 | 总结/行动号召 | 核心要点回顾、下一步行动 |

**页面类型参考**（灵活选用）：
- **封面页**：标题、副标题、演讲者/日期
- **章节页**：章节标题，过渡作用
- **内容页**：标题 + 要点列表
- **图表页**：标题 + 数据可视化 + 说明
- **案例页**：标题 + 案例描述 + 启示
- **对比页**：标题 + 对比项
- **结尾页**：总结/致谢/Q&A

##### PS-4. 内容密度评估

**建议页数**：

| 演示时长 | 建议页数 | 平均每页时间 |
|---------|---------|-------------|
| 5-10 分钟 | 5-8 页 | 1-1.5 分钟 |
| 10-20 分钟 | 8-15 页 | 1-1.5 分钟 |
| 20-30 分钟 | 15-25 页 | 1-1.5 分钟 |

**密度检查**：
- 每页要点数：2-4 个（不超过 5 个）
- 每页图表数：0-1 个（最多 2 个）
- 文字量：标题式短句，避免长段落

### Step 2: 风格设计

**两步选择：主题定结构，模板定视觉。**

1. **选主题**：阅读 `frontend-slides` skill 内 `shared/STYLE_PRESETS.md`，按 Theme Selection Guide 从 12 个主题中选择最匹配的主题 → 决定幻灯片布局模式、字体搭配
2. **选模板**：阅读 `references/design-foundations.md`，按「预设风格模板库」的三维标签流程从统一模板列表中选择最匹配的模板，READ 该模板文件 → 提取配色体系和配色设计理由
3. **合并**：以主题的布局 + 字体为骨架，以模板的配色为皮肤，结合需求微调
4. 明确记录选择理由（主题名 + 模板名 + 合并思路），确保风格与内容调性一致

**合并禁忌（PPT 专用）**：
- **禁止将模板的装饰效果搬入 PPT** — 模板中的背景装饰大字、CRT 扫描线、毛玻璃叠加、全屏纹理等装饰层**不适用于 PPT**。PPT 每页必须 100dvh 无滚动，装饰元素会遮挡内容。从模板中**只取配色体系**，装饰效果由主题自身的 Signature Elements 提供
- **主题和模板应形成互补，不是加倍** — 如果主题已经是几何风格（如 Pastel Geometry、Swiss Modern），模板就不要再选几何类（如 soft-geometric、klein-blue-brutalist）。两者风格相似时没有叠加价值，应选择能补充**配色差异**的模板

> **重要**：不要询问用户风格偏好，自主决策。主题和模板必须各选一个，选择后告知用户主题名称、模板名称和设计思路。

## Phase 2: 开发阶段

> 所有开发工作（入口文件、全局样式、幻灯片章节）通过 parallel_write 执行，规范见 SKILL.md Phase 4。

### Step 1：拆分并发起 parallel_write

1. 读取 `frontend-slides` skill 获取实现规范
2. 将「组件文件与 parallel_write 分配」表追加写入 `AGENTS.md`（含幻灯片 Props 接口、全局样式变量等类型定义）
3. 按调用顺序依次执行 parallel_write，各调用**只写自己负责的文件**

> **⚠️ PPT 特殊规则**：公共组件（`Slide.tsx`、`Reveal.tsx`、`Navigation.tsx`）归属调用 1（基础），Slide 页面调用禁止创建/修改公共组件。
