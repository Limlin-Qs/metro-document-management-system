# 应用场景开发规范（prototype）

> **本文档是应用场景（单页/多页）的权威开发流程规范。** 严格按照下方定义的两个阶段逐步执行，不可跳过或合并任何步骤。

---

## 场景约束

- **页面模式**: 单页应用（默认）或多页应用，根据需求复杂度决定
- **布局**: 各区块按独立 Section 组件拆分，多页应用时每个页面为独立路由
- **导航**: 根据页面模式选择导航方式（见下方导航策略）

**页面模式判断**：

| 页面模式 | 适用场景 | 典型案例 |
|---------|---------|---------|
| **单页** | 内容可在一页内展示，功能单一或线性浏览 | 官网落地页、产品介绍、计算器、单页表单 |
| **多页** | 存在多个独立功能区域，内容无法合理塞入一页 | 管理后台、博客（首页+详情）、电商（商品列表+详情+购物车） |

> 拿不准时默认单页。

**导航响应式**（含导航时强制）：
- 宽屏（≥768px）：展示完整水平导航
- 窄屏（<768px）：使用 Hamburger 折叠菜单或将导航项收纳为下拉
- 禁止：窄屏下导航与内容争夺水平空间

### Step 1: 需求分析

<thinking>

##### AP-1. 需求理解

从用户消息中提取：
- **产品类型**：应用/网站的分类（官网、博客、工具、落地页、教程等）
- **核心功能**：主要功能模块和使用场景
- **目标用户**：用户画像和核心诉求
- **界面语言**：参照通用判断规则

##### AP-2. 导航策略

**判断导航模式**（与页面模式联动）：

| 导航模式 | 页面模式 | 适用场景 | 典型案例 |
|---------|---------|---------|---------|
| **路由导航** | 多页 | 多个独立页面，需在页面间切换 | 管理后台、博客、电商 |
| **锚点导航** `#anchor` | 单页 | 多区块内容，需要快速定位 | 官网、产品介绍页 |
| **无导航** | 单页 | 功能单一，无需跳转 | 计算器、单页表单、小工具 |

**路由导航时，规划路由结构**：
- 列出所有页面及其路由路径（如 `/`, `/about`, `/blog/:id`）
- 导航项对应核心页面，建议 3-7 个
- 区分固定导航项和动态路由（如详情页不出现在导航中）

**锚点导航时，规划导航项**：
- 只放核心区块入口，建议 3-6 个导航项
- 导航项与页面区块一一对应

##### AP-3. 页面与区块规划

**单页应用**：将内容拆分为有序区块。
**多页应用**：先规划页面列表，再对每个页面拆分区块。

**规划原则**：
- 每个区块承担一个明确职责
- 区块之间有逻辑关系（递进/并列/对比）
- 单页时：首屏（Hero）吸引注意，末尾（CTA/Footer）引导行动
- 多页时：每个页面有明确的独立职责，页面间通过路由跳转；跨页面复用的区块提取为共享组件

**常见区块类型**：

| 区块类型 | 适用内容 | 布局特点 |
|---------|---------|---------|
| Hero/首屏 | 标题 + 副标题 + CTA | 大字号，视觉冲击 |
| Feature Grid | 功能/特性展示 | 2-4 列网格 |
| 内容段落 | 详细说明/介绍 | 文字为主，舒适阅读 |
| 卡片列表 | 文章/项目/案例展示 | 网格排列 |
| 统计数字 | 关键数据展示 | 大数字 + 标签 |
| CTA 区块 | 行动号召 | 居中，背景色区分 |
| 表单区域 | 用户输入 | 居中，限制宽度 |
| Footer | 链接/版权信息 | 全宽背景 |

##### AP-4. 功能拆解

**功能点**：3-8 个核心功能点（多页应用按整体统计，非每页单独计）

**规则**：
- 不列出通用交互（如"按钮hover"、"表单验证"）
- 不将导航列为功能点（导航是全局配置，已在导航策略中规划）
- 信息不足时基于合理推断补充

#### AP-D1. 理解需求，确立美学方向

**需求提取**：
- 从 Step 1 需求分析中提取：应用类型、目标用户、核心功能、风格关键词
- 识别产品需要传递的情感，用 1-2 个关键词描述核心调性

**选择美学方向**：说明该风格的哪些特征能传达目标调性，以及应避免的视觉特征。

**禁止组合**：儿童产品 × 暗黑恐怖 / 医疗产品 × 高刺激霓虹 / 金融 × 过度活泼

**选择风格模板**：从 design_foundations「预设风格模板库」中选择最匹配的模板，READ 模板文件，提取其设计参数（配色、字体、布局、装饰手法），然后结合需求调整。

**输出**：参考模板名称 + 核心调性 + 美学方向 + 调整说明

#### AP-D2. Typography（字体）

根据应用场景、品牌人格和用户情绪选择字体组合：

| 场景调性 | 中文方向 | 英文方向 |
|---------|---------|---------|
| 严肃企业/金融 | 思源黑体 Bold、思源宋体 | Inter, Roboto, Playfair Display |
| 科技/极简/未来感 | 思源黑体 ExtraLight、苹方 | Montserrat, Space Grotesk |
| 活泼/创意 | 圆体、手写感体 | Poppins, Fredoka One |
| 文化/典雅 | 楷体、思源宋体 | Lora, Garamond |

要求：层级清晰（Heading vs Body）、双语和谐、优先 Web 安全字体或 Google Fonts。

#### AP-D3. Color & Theme（色彩与主题）

- **主导色 + 锐利强调色** 优于平均分布的多色方案
- 避免纯黑/纯白，使用 Off-white 或 Tinted Black
- 有色背景必须指定文字颜色，对比度 >= 4.5:1（大文本 >= 3:1）
- 从参考模板提取基础配色，结合需求调整

#### AP-D4. Spatial Composition（空间构成与布局）

根据应用类型推导布局策略：

| 应用类型 | 布局策略 | max-width 参考 |
|---------|---------|---------------|
| 营销/落地页 | 全宽流式，强调视觉冲击力 | 72rem ~ 80rem |
| 博客/内容站 | 舒适阅读宽度 | 48rem ~ 64rem |
| 工具类 SPA | 聚焦核心区域，移除干扰 | 28rem ~ 48rem |

**导航模式**（根据 Step 1 导航策略）：
- **锚点导航**: 固定顶部，禁止透明背景，高度 56~64px，Logo 左/导航项右
- **无导航**: 聚焦核心功能区，无干扰元素

**全局间距契约**：
- 全局容器提供响应式 padding，页面内容填充容器，禁止自设外边距
- 区块间距统一（`padding: 4rem 0` ~ `padding: 6rem 0`）
- 全宽背景策略：Hero、CTA 可使用全宽背景色，但内容仍受 max-width 约束

#### AP-D5. Motion & Visual Effects（动效与视觉效果）

- 从参考模板继承动效风格，说明具体动画属性（transform/opacity/scale）和时长
- 每个可交互元素需有 hover/active 反馈
- 圆角、阴影参数从所选 style-preset 文件中提取

**复杂背景文字处理**：

| 背景类型 | 处理方案 |
|---------|---------|
| 渐变背景 | 以最浅区域计算对比度，不达标时添加文字阴影或半透明底板 |
| 图片背景 | 必须添加遮罩层 `rgba(0,0,0,0.4~0.6)` 或文字投影 |
| 有色背景 | 明确指定文字颜色 HSL 值，禁止模糊描述 |

#### AP-D6. 组件状态设计

核心组件需覆盖 Default / Hover / Focus / Disabled 状态：

- **按钮**: Primary（背景 primary / Hover 降低不透明度）、Secondary（背景 surface / 边框）、Ghost（透明 / Hover accent 背景）
- **表单元素**: 输入框（背景 surface / Focus primary 描边）、Placeholder（textMuted 色）
- **卡片**: 背景 surface / 阴影 / Hover 阴影增强或微位移

## Phase 2: 开发阶段

> 所有开发工作通过 parallel_write 执行，规范见 SKILL.md 全局约束中的 parallel_write 工具规范。

### Step 1：拆分并发起 parallel_write

1. 将「组件文件与 parallel_write 分配」表追加写入 `AGENTS.md`（含区块 Props 接口、路由/锚点 ID 等类型定义）
2. 按调用顺序依次执行 parallel_write，各调用**只写自己负责的文件**

**拆分粒度规则**：

| 调用顺序 | 层级 | 说明 |
|---------|------|------|
| 第 1 次 | 基础设施 | 主题变量、全局样式、导航/Layout 组件、共享组件（多页时） |
| 第 2 次 | 业务组件 | 所有 Section 组件合并到一次调用，并发写入 |
| 第 3 次（最后） | 入口+路由 | 所有页面文件 + `app.tsx`（import 所有组件，必须最后执行） |

**示例 A：单页产品官网（锚点导航，5 个区块）**

```
## 组件文件与 parallel_write 分配
| 调用 | 文件路径 | 说明 |
|------|---------|------|
| 调用 1（基础） | `index.css`, `components/layout.tsx` | 主题变量+全局样式+导航 |
| 调用 2（业务组件） | `HomePage/components/HeroSection.tsx`, `HomePage/components/FeaturesSection.tsx`, `HomePage/components/TestimonialsSection.tsx`, `HomePage/components/PricingSection.tsx`, `HomePage/components/ContactSection.tsx` | 所有区块组件并发写入 |
| 调用 3（入口+路由） | `HomePage/HomePage.tsx`, `app.tsx` | 页面组装+入口（最后执行） |
→ 3 次串行调用，每次调用内的文件并发写入
```
