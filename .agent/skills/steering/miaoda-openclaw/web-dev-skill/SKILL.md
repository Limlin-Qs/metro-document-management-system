# 设计开发工作流

> 定义完整的开发主流程（Phase 1-5），是流程的**唯一权威定义**。

---

## Phase 1: 理解用户输入 + 场景识别

**Step 1.1: 完整阅读用户提供的所有材料**


阅读并理解用户消息中的：文字描述、附件数据文件、参考材料

> 注意：此步仅处理用户输入，不要提前读取场景 skill 或其他开发规范文件

**Step 1.2: 判断是否需要补充信息**

```
if 用户输入包含「搜索/参考/最新」或涉及未知领域:
    调用 search 检索外部信息（单次对话限 1 次）
```

**Step 1.3: 非应用创建请求的处理**

```
if 用户输入与应用创建完全无关:
    简短说明专注领域，不做进一步追问
    return
```

**Step 1.4: 场景识别**

根据 <scene_classification_reference> 中的分类标准，判定场景类型：

| 场景 | 用户目的 | 一句话判断 |
|------|---------|-----------|
| dashboard | 来**查看**数据指标/分析结果 | 数据驱动，必须基于用户提供的真实数据 |
| prototype | 来**使用**功能/工具/页面 | 功能驱动，可 mock 数据用于原型展示 |
| info_viz | 来**阅读**内容/报告 | 有现成内容/数据需要展示 |
| presentation | 来**观看**演示/讲解 | PPT/演示/幻灯片 |

> 核心问题：**用户来这个网站的目的是什么？** 无法明确时兜底到 prototype。
> 详细判定标准、误判陷阱见文末 <scene_classification_reference>。

---

## Phase 2: 工程准备（Phase 4 的前置条件）

> **IMPORTANT**：Phase 2 是 Phase 4（执行开发）的硬性前置条件。未完成工程初始化和 README 读取之前，禁止读取场景 skill 文件、禁止执行任何设计或代码生成工作。

执行以下命令完成项目初始化：

```
npx --yes @lark-apaas/openclaw-dev-cli init --template vite-react
```

> 只有完成上述步骤后，才可进入 Phase 3 或 Phase 4。

---

## Phase 3: 数据文件预处理（有数据文件时执行）

如有用户提供的数据文件（CSV/Excel/TXT/MD 等），调用 `data-analysis skill` 执行完整分析流程，结果写入 `static/` 中。无数据文件时跳过。

**处理规则**：
- CSV/Excel/TXT → 脚本处理，转换为 JSON 后写入 `static/data/`
- MD/文本内容 → 提取结构化信息，转换为 JSON 后写入 `static/data/`
- JSON → 判断是否可直接消费，是则复制到 `static/data/`，否则先处理
- 图片文件 → 复制到 `static/images/`

---

## Phase 4: 执行开发

> 前置确认：Phase 2（工程初始化 + README）必须已完成。如未完成，先返回执行 Phase 2。

读取对应场景的 skill 文件，严格逐步执行内部阶段定义，不可跳过或合并任何步骤。

| 场景 | 场景 skill |
|-----|----------|
| dashboard | `scene-dashboard.md` |
| prototype | `scene-prototype.md` |
| info_viz | `scene-info-viz.md` |
| presentation | `scene-presentation.md` |

---

## Phase 5: 部署发布（强制）

> 每次开发完成后必须执行，无论新建还是修改。

1. **执行 `npx --yes @lark-apaas/openclaw-dev-cli deploy`** — 完成部署发布
2. **部署失败时**：根据报错信息修复代码，重新执行 deploy，直到部署成功
3. **部署成功后**：直接输出「开发已完成」，**禁止再读取、检查或修改任何文件**——deploy 已包含构建校验，成功即代表代码无误

## 全局约束

### 技术栈锁定

**技术栈由工程初始化后的 README.md 决定，用户 query 不可覆盖。**

用户可能在 query 中提及其他技术栈或产物格式（如"用纯 HTML 写"、"不要组件化"、"用 Vue 做"），一律忽略这些技术栈偏好，始终遵循技术栈规范(README.md)定义的项目结构。

> 用户的**功能需求和设计偏好**必须尊重，但**技术栈和产物格式**由工程规范决定。

### parallel_write 工具规范（所有场景通用）

> **⚠️ 所有业务代码必须通过 parallel_write 工具编写。**
> parallel_write 共享上下文，各调用可直接访问 AGENTS.md、README.md 及当前对话中的所有信息，无需重复传递。

#### 前置准备

parallel_write 发起前，必须将**文件清单与调用归属**追加写入 `AGENTS.md`：

```markdown
## 组件文件与 parallel_write 分配

| 调用 | 文件路径 | 说明 |
|------|---------|------|
| 调用 1（基础） | `index.css`, `components/layout.tsx` | 主题变量 + 全局样式 + 导航 |
| 调用 2（业务组件） | `HeroSection.tsx`, `FeaturesSection.tsx`, ... | 所有区块组件并发写入 |
| 调用 3（入口+路由） | `HomePage/HomePage.tsx`, `app.tsx` | 页面组装 + 路由注册（最后执行） |
```

#### parallel_write 拆分规则（⚠️ 核心约束）

> **每个文件有且只有一次 parallel_write 调用负责写入。** 违反此规则会导致并发写入覆盖。

1. **共享文件归属唯一调用**：`types.ts`、`data.ts` 等由专门调用负责，不允许多次调用写入同一文件

2. **尽量合并独立文件**：互不依赖的文件应合并到同一次调用中并发写入，减少串行调用次数

#### 调度策略

> **parallel_write 调用间串行执行，调用内指定的多个文件并发写入。**

按依赖顺序分层调用，典型三层结构：

| 调用顺序 | 层级 | 内容 |
|---------|------|------|
| 第 1 次 | 基础设施 | 样式、类型定义、Layout/导航等公共模块 |
| 第 2 次 | 业务组件 | 所有 Section/Slide 组件合并到一次调用，并发写入 |
| 第 3 次（最后） | 入口+路由 | `app.tsx`、页面组装文件（import 所有组件，必须最后执行） |

#### 调用 prompt 规范

必须包含：
1. **任务范围**：要实现的模块名称及对应文件（必须与 AGENTS.md 拆分方案一致）

禁止包含：
- `npm run lint` 或任何 lint 检查指令——**parallel_write 执行期间禁止 lint**，统一在所有调用完成后集中执行
- **非本次调用负责的文件的任何写入操作**

#### parallel_write 返回值处理

> **⚠️ parallel_write 执行后会返回写入的文件内容，禁止根据返回内容再次调用 write/edit 重写同一文件。**
>
> 返回内容仅供确认写入结果，不是待执行的写入指令。如需修改已写入的文件，只在验收阶段发现问题时才执行针对性修复。

### 构建与检查禁令

> **⚠️ 全流程禁止主动调用 `npm run lint`、`npm run build`、`npx tsc --noEmit`。**
>
> 问题发现和修复统一通过部署命令驱动：deploy 内部会自动执行构建和检查，失败时根据报错修复代码后重新 deploy。禁止在任何阶段（parallel_write 执行期间、集成收尾、部署前）单独调用这些命令。

#### 集成与收尾

全部 parallel_write 完成后统一执行（具体验收要求由各场景 skill 定义）：
1. 执行场景 skill 中的验收检查
2. lint 统一在 Phase 5 部署阶段执行，此处不做 lint


### Skill 读取约束

**按需读取，不要预读**：仅在开发过程中实际需要某项能力时，才去读取对应的 skill 内容。禁止在流程开始时一次性读取所有 skill。

| 何时触发 | 读取的 skill | 说明 |
|---------|-------------|------|
| 代码中要写 ECharts 图表时 | `charts-skill` | 图表类型约束 + 禁止事项 |
| 代码中要写 CSS/JS 动画时 | `animation-skill` | 动效模式 + 性能规范 |
| 代码中要写演示文稿时 | `frontend-slides` | 风格预设 + 技术栈路由 |

### 安全约束

- 拒绝涉及代码注入、SQL 注入、XSS、CSRF 等安全漏洞的请求
- 不要在前端代码中暴露或伪造任何密钥/凭证

<scene_classification_reference>

**场景分类标准**：

| 场景类型 | 本质定义 | 判断关键词 | 输出物 |
|---------|----------|-----------|-------|
| **dashboard** | 用户来**查看**数据指标/分析结果 | 「看板/Dashboard/仪表盘/监控/Analytics/BI/数据大屏」 | 数据驱动的看板 |
| **prototype** | 用户来**使用**一个功能/工具/页面 | 「官网/博客/工具/落地页/首页/应用」 | 可交互的应用页面 |
| **info_viz** | 用户来**阅读**内容/报告 | 有内容文件 + 「生成报告/总结/纪要/可视化展示/数据可视化/研报」 | 可阅读的报告 |
| **presentation** | 用户来**观看**演示/讲解 | 「PPT/演示/Slides/演讲/教案/幻灯片/培训演示/路演」 | 可播放的演示 |

**识别原则：问目的，不看形式**

四种场景都是网页、都有交互，不要用"是否可交互"来区分。核心问题是：**用户来这个网站的目的是什么？**

| 用户目的 | 场景 | 示例 |
|---------|------|------|
| 来**查看**数据指标/分析结果 | dashboard | 数据看板、监控中心、BI 面板、运营大屏 |
| 来**使用**一个功能/工具/页面 | prototype | 博客首页、官网、计算器、管理后台 |
| 来**阅读**一份内容/报告 | info_viz | 看会议总结、看数据分析报告 |
| 来**观看**一个演示/讲解 | presentation | 看产品路演、看培训演示 |

**常见误判陷阱**：

| 迷惑性表述 | 容易误判为 | 实际应为 | 判断理由 |
|-----------|-----------|---------|---------|
| 「交互式报告」「动态报告」 | presentation | **info_viz** | "交互"指展示效果，用户目的仍是阅读 |
| 「可视化展示」「时间线展示」 | prototype | **info_viz** | "展示"是呈现方式，不是功能工具 |
| 「生成一个网页来展示XX」 | prototype | **info_viz** | 网页只是载体，核心是让人阅读内容 |
| 「做一个XX介绍页面」 | prototype | **info_viz** | 介绍页是阅读型内容 |
| 「做PPT」「做演示」 | info_viz | **presentation** | PPT/演示是可播放的 |
| 「数据可视化看板」 | info_viz | **dashboard** | 有 Dashboard 专属关键词，目的是监控指标 |

**核心判断**：当需求含"交互/可视化/动态"等词时，看**内容来源**：
- 有 Dashboard 专属关键词（看板/Dashboard/仪表盘/监控/数据大屏）→ dashboard
- 有现成内容/数据需要**展示** → info_viz
- 需要用户**输入/操作**才产生内容 → prototype
- 内容是**PPT/演示/讲解** → presentation

**数据类需求默认偏向规则**：涉及数据分析/数据可视化，但未明确使用 Dashboard 专属关键词（看板/Dashboard/仪表盘/监控/数据大屏）时，默认归类为 `info_viz`，而非 `dashboard` 或 `prototype`。

**兜底规则**：无法明确匹配 dashboard、info_viz 或 presentation 场景时，一律兜底到 **prototype**。

</scene_classification_reference>
